Moonlander was created in 1973 and if you weren't there you might be amazed to learn that computer technology was a bit more primitive and software development was a bit more difficult.    Here are some interesting facts about the Moonlander software, hardware, and development environment.

Q.    What was the hardware used to play Moonlander?

A.    Moonlander was written and played on a Digital Equipment Corporation (DEC) PDP‑11 minicomputer attached to a VT11 vector graphic display.  When combined in a single chassis the processor/display combination was called a GT40.

        The GT40 had a 12Ó display and a viewable resolution of 1024 x 768 (the same as the original iPad) and used a light pen to interact with the display.

Q.    How much did the Moonlander hardware cost?

A.    It cost about $15,000 purchase a GT40 graphic terminal back in 1973 (around $75,000 in 2013 dollars).  If you played this game you were likely working in industry or in a college or university as the personal computer had yet to be invented.

Q.    How big was the original Moonlander application?

A.    The PDP‑11 binary file is about 13KB in size.  Moonlander was designed for a 16‑bit computer with just a total address space of 64KB.  For comparison your iPad has at least 512MB of memory to run applications, about 8000x more memory than the GT40 graphic terminal.

Q.    What is different in this modern version of Moonlander?

A.    In the original Moonlander, the game repeats forever, causing the crashed landers and flags to litter the lunar surface.  In order to give the user an option to end the game, a dialog was added so you could quit and return to the menu screen (you can also do this at any time by shaking the iPad).  The original hardware used a light pen so the controls have been slightly enlarged and spread out to make it easier to control the lander.  Finally, the fonts been increased in size since the iPad display is a bit smaller than the original.

Q.    What are some of the original bugs in the classic version of Moonlander?

A.    One bug is that the original Moonlander kicks up dust when you are out fuel:

;
;               THIS ROUTINE WILL KICK UP DUST ON THE MOON.
;
DUST:   CMP     RADARY,#150.            ;SEE IF WE ARE CLOSE TO THE MOON NOW.
        BGE     NODUST                  ;DON'T GENERATE ANY DUST NOW.
        MOV     #EXLIST,TEMP2           ;PICK UP ADDRESS OF WHERE TO LEAVE DUST.
        MOV     PERTRS,TEMP             ;GET THE TRUST NOW.
        CMP     TEMP,#63.               ;SEE IF MORE THAN 63%
        BLE     DUSTB1                  ;NO. OK.
        MOV     #63.,TEMP               ;IF MORE, SET TO 63% FOR SCALING.

        If the fuel has run out your thrust drops to zero but the original only looks at the percent thrust selected on the slider which continues to operate until you land or crash.

Q.    Any other interesting bugs?

A.    Legend has it there are other well known bugs but you should explore the game and try to find them yourself.  The modern version does not fix any bugs so go ahead and see if you can get to the other side of the mountain.

Q.    What were some of the challenges in recreating Moonlander?

A.    Learning how to program an PDP‑11 minicomputer in assembly language was one of the bigger challenges.  Sometimes you end up staring for a long time at code like this:

        CLR     #MACTHR                 ;TOO CLOSE. WE'VE CRASHED INTO IT.
        .=.-2                           ;AND LIKEWISE DEFINE THE MAC DESTROYED
MACTHR: .WORD   MACTHR                  ;FLAG AS STILL HAVING MACDONALDS.

        or my favorite example of self modifying code:

DRAW22: INC     @#DFUDGE                ;---<<INSTRUCTION CHANGED>>---
        CMP     DFUDGE,#3.
        BLT     DRAW2V
        MOV     (PC)+,@(PC)+            ;WTF? (my comment)
        DEC     @(PC)+
        .WORD   DRAW22
        BR      DRAW2W
DRAW2V: CMP     DFUDGE,#-3.

        The other challenge was trying to recreate Moonlander from memory (having last played in the mid-1970Õs) and the original source code.  This would not have been possible without the work of the simh project and the PDP‑11 and VT11 simulators and the work of computer historians like bitsavers.org.
